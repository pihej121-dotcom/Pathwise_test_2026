Add a new tool to ChatHome called "Mock Interview" (a video interview assistant). It asks interview questions aloud, records the user's video answers, transcribes them, and critiques both WHAT they said (content) and HOW they said it (delivery).

This is a larger feature than the other tools. Before writing any code, read the plan below and report back your approach, which APIs/services you'll use, and any new costs or keys required. Do not write code until I confirm.

Question source:
- Reuse the interview questions generated by the existing Interview Prep flow. Do not rebuild question generation. The Mock Interview takes those questions and presents them one at a time.

Per-question flow:
1. Ask the question aloud using the browser's built-in Web Speech API (SpeechSynthesis) for text-to-speech. Free, no paid voice service for now. Also show the question text on screen.
2. Record the user's answer via webcam + mic using the browser MediaRecorder API (getUserMedia). Show a live camera preview, and start/stop recording controls.
3. After each answer, transcribe the recorded audio using a speech-to-text service (OpenAI Whisper or equivalent). This is the one expected paid dependency. Flag the key/cost in your plan.
4. Move to the next question.

Critique (after the session, or per answer):
Send the transcript plus timing data to the existing AI endpoint to critique:
- CONTENT: relevance, completeness, use of structure (e.g. STAR), specificity, honesty about weak answers.
- DELIVERY (transcript-based, reliable): speaking pace (words per minute), filler words ("um", "like", "you know") with counts, rambling vs conciseness, answer length.
- DELIVERY (audio tone, best-effort only): derive what is realistically possible from the audio and timing, such as long pauses, pacing variation, and energy/volume proxies. Do NOT claim to detect emotions like confidence or nervousness unless you can do so reliably. In the output, be honest that tone analysis is approximate. If robust audio-tone analysis isn't feasible without an expensive service, say so in your plan and propose the lighter version instead.

Constraints:
- Do NOT do video/computer-vision analysis (no posture, eye contact, facial expression). Out of scope.
- Browser permissions: handle camera/mic permission denial gracefully with a clear message.
- Consistent with the other flows: session state only, no new database persistence. Save the conversation to localStorage via saveSession for logged-in users.
- Format critique output using the same markdown conventions the other tools use so it renders in MessageContent.
